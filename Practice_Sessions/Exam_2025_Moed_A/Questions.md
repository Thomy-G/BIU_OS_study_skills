## שאלה 3 (20 נקודות) – מניעת חסימות (Deadlock Proofs)

נתונה מערכת עם $m$ יחידות זהות של משאב מסוים ו-$n$ תהליכים. כל תהליך יכול לבקש או לשחרר יחידה אחת בכל פעם, ויש לו צורך מקסימלי של בין $1$ ל-$m$ יחידות משאב.  
ידוע כי סכום כל הצרכים המקסימליים (ה-Max Need) של כל התהליכים ביחד קטן מ-$m+n$, כלומר:
$$\sum_{i=1}^{n} Max\_i < m + n$$

להלן ניסיונות להוכיח האם במערכת יכול להתרחש Deadlock או שבהכרח לא יכול להתרחש Deadlock. קבע עבור כל הוכחה אם היא **נכונה** או **שגויה**. במידה והיא שגויה, נמק במשפט אחד.  
*(שימו לב: נימוק שגוי להוכחה לא נכונה לא מזכה בנקודות כלל).*

* **Proof A:** נניח Deadlock קורה. אזי כל $m$ המשאבים נמצאים בשימוש. מכיוון ש-$\sum Max\_i < m+n$ הרי שבהכרח $\sum Need\_i = \sum Max\_i - m < n$. כלומר לפחות תהליך אחד חייב להחזיק יותר ממשאב אחד וברגע שאותו תהליך ישחרר את המשאבים שהוא מחזיק יוכלו תהליכים אחרים להתקדם וכו'. מכאן ש-Deadlock אינו אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** המסקנה מבוססת על הנחה לא תקפה – החזקת יותר ממשאב אחד אינה מרמזת שתהליך יכול לשחרר אותו באופן יזום או עצמאי באמצע תהליך ה-Deadlock ללא קבלת שאר המשאבים להם הוא זקוק.

* **Proof B:** נניח שמתקיים Deadlock. במצב זה כל ה-$m$ משאבים מוקצים ($\sum Allocation\_i = m$), ומכאן: $\sum Need\_i = \sum Max\_i - m < n$. ומכאן שקיים תהליך שעבורו $Need\_i = 0$, אולם מאחר ולכמה תהליכים יש Need הגדול מאפס, יכול להיות מעגל בינם לבין עצמם. מכאן שיכול להתקיים Deadlock.  
  **קביעה:** **שגויה**  
  **נימוק:** ההוכחה סותרת את עצמה; התהליך שעבורו $Need\_i = 0$ סיים את דרישותיו, ולכן הוא יחזיר למערכת את המשאבים שהוא מחזיק ונחזור למצב התחלתי עם סכום כולל קטן יותר של Need, מה שמונע חסימה מעגלית.

* **Proof C:** במידה וכל תהליך מבקש את המקסימום האפשרי שלו, הרי שסה"כ הדרישה יכולה להגיע ל-$m+n-1$, ומכיוון שסה"כ המשאבים במערכת הוא $m$, ואין מניעה ש-$m>n$, הרי שייתכן וכל אחד מהתהליכים מחזיק כמות מסוימת של משאבים ומבקש את ה-Max שלו, והכמות שנשארה Available היא 0 (או קטנה מה-Need של כל אחד מהתהליכים). ומכאן ש-Deadlock אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** אין קשר לוגי או מתמטי בין התנאי $m>n$ להמשך הטיעון במשפט.

* **Proof D:** מכיוון שכל תהליך זקוק לכל היותר ל-$m$ יחידות, וישנן $m$ יחידות בסך הכל, כל תהליך יכול בסופו של דבר לקבל את כל מה שהוא צריך. לכן, Deadlock אינו אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** העובדה שכל תהליך יכול תיאורטית לקבל את המקסימום לבדו אינה מונעת תנאי המתנה בו-זמניים של תהליכים שונים המגדירים מצב Deadlock.

* **Proof E:** נניח שמתקיים Deadlock. אזי כל המשאבים מוקצים, כלומר $\sum Allocation\_i = m$. מכאן ש-$\sum Need\_i = \sum Max\_i - m < n$, כלומר קיים לפחות תהליך אחד שעבורו $Need\_i = 0$. תהליך זה יסיים את פעולתו וישחרר את המשאבים שהוא תופס, וכך הלאה לגבי יתר התהליכים בצורה סדרתית, מה שמוביל לסתירה ולכן Deadlock לא יכול להתרחש.  
  **קביעה:** **נכונה**

* **Proof F:** נניח שמתקיים Deadlock. במצב זה כל המשאבים מוקצים. מכאן $\sum Need\_i = \sum Max\_i - m < n$, כלומר, כל תהליך צריך פחות מיחידה אחת נוספת, כלומר אין אף תהליך שממתין למשאב. הדבר מוביל לסתירה ולכן Deadlock לא יכול להתרחש.  
  **קביעה:** **שגויה**  
  **נימוק:** ההסקה שאם הממוצע קטן מ-1 אז *כל* תהליך צריך פחות מיחידה אחת איננה נכונה מתמטית ולוגית (ייתכן שחלק צריכים 0 וחלק צריכים יותר מ-1).

* **Proof G:** נגדיר את ה-Worst Case כמצב בו כל תהליך מחזיק בכמות יחידות משאב השווה ל-Maximum שלו פחות אחד ($Allocation\_i = Max\_i - 1$). מכאן:
  $$\sum Allocation\_i = \sum (Max\_i - 1) = \sum Max\_i - n$$
  מכיוון שנתון $\sum Max\_i < m+n$, מתקיים ש-$\sum Max\_i - n < m$, ולכן $\sum Allocation\_i < m$.  
  כלומר, אפילו במצב הגרוע ביותר לא כל יחידות המשאב מוקצות לתהליכים, תמיד יש לפחות משאב אחד פנוי ($Available \ge 1$), ולכן קיים תהליך שיכול לקבל עוד יחידת משאב, לסיים ולשחרר משאבים לטובת האחרים. ומכאן ש-Deadlock לא יכול לקרות.  
  **קביעה:** **נכונה**

* **Proof H:** מכיוון שכל תהליך מבקש יחידת משאב אחת בכל פעם, המערכת תמיד תוכל להתקדם. לכן Deadlock אינו אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** הנחה זו מניחה התקדמות מבלי להוכיח שלא תיווצר המתנה מעגלית; בקשת יחידה אחת בכל פעם כשלעצמה אינה מונעת Deadlock.

* **Proof I:** במצב בו $\sum Max\_i < m+n$ יכול להתרחש Deadlock מכיוון שכל תהליך יכול לבקש יחידת משאב אחת בלבד ולהחזיק בה. במידה וכל תהליך גם יבקש יחידה נוספת, נוצרת המתנה מעגלית ואף אחד לא יכול להמשיך.  
  **קביעה:** **שגויה**  
  **נימוק:** במצב המתואר מוחזקות $n$ יחידות ומתבקשות עוד $n$ יחידות, אך מכיוון ש-$\sum Max\_i < m+n$ ובהכרח $m > n$ לפי נתוני השאלה, יישארו משאבים פנויים במערכת שניתן להקצותם, ולכן המעגל לא ייחסם.

* **Proof J:** מכיוון ש-$\sum Max\_i < m+n$, ויש לנו $m$ יחידות מהמשאב, אז בכל עת יש מספיק משאבים כדי לספק לפחות תהליך אחד. לפיכך, Deadlock לא יכול להתרחש.  
  **קביעה:** **שגויה**  
  **נימוק:** המסקנה מניחה באופן שגוי שביקוש כולל היפותטי מבטיח זמינות של משאבים פנויים בכל רגע נתון, תוך התעלמות מאופן ההקצאה הדינמית בפועל.

### 4 & 5. ניתוח קוד סיגנלים וצינורות (Signals & Pipes)
בהינתן קוד המקור הבא:

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
#include <signal.h>
#include <stdlib.h>

void handler_usr1(int sig) {
    printf("Handler SIGUSR1: Signal %d received in process %d\n", sig, getpid());
}
void handler_usr2(int sig) {
    printf("Handler SIGUSR2: Signal %d received in process %d\n", sig, getpid());
}
int main() {
    int pipefd[2];
    pipe(pipefd);
    signal(SIGUSR1, handler_usr1);
    
    pid_t pid1 = fork();
    if (pid1 == 0) { // בן ראשון
        close(pipefd[1]);
        char buf;
        read(pipefd[0], &buf, 1);
        close(pipefd[0]);
        kill(getpid(), SIGUSR1);
        return 0;
    }
    
    signal(SIGUSR2, handler_usr2);
    pid_t pid2 = fork();
    if (pid2 == 0) { // בן שני
        close(pipefd[0]);
        kill(getppid(), SIGUSR2);
        write(pipefd[1], "V", 1);
        close(pipefd[1]);
        return 0;
    }
    
    close(pipefd[0]);
    close(pipefd[1]);
    pause();
    waitpid(pid1, NULL, 0);
    printf("Parent process %d done!\n", getpid());
    return 0;
}
```

**נתונים והנחות:**
* הורה (Parent) PID = 20.
* בן ראשון (Child 1) PID = 21.
* בן שני (Child 2) PID = 22.

#### שאלה 4: מה תהיה השורה הראשונה שתודפס בעת הרצת התוכנית?
* א. `Handler SIGUSR1: 10 received in process 20`
* ב. `Handler SIGUSR1: 10 received in process 21`
* ג. `Handler SIGUSR2: 12 received in process 20`
* ד. `Handler SIGUSR2: 12 received in process 22`  
**תשובה:** `_____`  
*הסבר:* הבן הראשון (21) חסום בקריאת `read` מהצינור כי הצינור ריק. הבן השני (22) רץ מיד ושולח `SIGUSR2` לאבא שלו (20) באמצעות `kill(getppid(), SIGUSR2)`. האבא שהיה ב-`pause()` מתעורר ומריץ את ה-Handler שלו, שמדפיס את השורה של `SIGUSR2` עבור תהליך 20. רק לאחר מכן הבן השני כותב לצינור ומשחרר את הבן הראשון.
