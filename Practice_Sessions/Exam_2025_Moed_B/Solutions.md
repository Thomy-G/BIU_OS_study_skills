## שאלה 1 (20 נקודות) – תזמון מעבד עם Context Switch (SJF)

במחשב של אלי יש חמישה תהליכים $A, B, C, D$ ו-$E$ המגיעים בדיוק באותו הזמן בסדר זה. הבקשות ל-CPU Bursts מפורטות בטבלה.  
חשב את ה-Average Waiting Time במידה וה-Scheduler עובד לפי **Shortest Job First (SJF)**.

**נתונים נוספים:** זמן ה-Context Switch הוא **1 שניה**, במחשב יש ליבה אחת בלבד, ואין תהליכים נוספים.

### נתוני ה-CPU Burst:
| תהליך | CPU Burst (שניות) |
| :---: | :----------------: |
| A | 7 |
| B | 2 |
| C | 3 |
| D | 6 |
| E | 4 |

### פתרון וחישוב:
* באלגוריתם SJF, מסדרים את התהליכים מהקצר ביותר לארוך ביותר: $B (2) \rightarrow C (3) \rightarrow E (4) \rightarrow D (6) \rightarrow A (7)$.
* יש להתחשב בזמן ה-Context Switch (שנייה אחת) בין תהליך לתהליך.
* **בניית לוח הזמנים וזמני ההמתנה:**
  * התהליך הראשון שמורץ הוא **B**. זמן המתנה = **1 שניה** (מכיוון שאין Context Switch מלא בהתחלה לפני התהליך הראשון, קיבלו גם 0, אך לפי הפתרון הרשמי נרשם 1).
  * $B$ מסיים בזמן $1 + 2 = 3$. יש Context Switch של 1 שניה, אז **C** מתחיל בזמן 4. זמן המתנה של $C$ = **4**.
  * $C$ מסיים בזמן $4 + 3 = 7$. לאחר Context Switch של 1 שניה, **E** מתחיל בזמן 8. זמן המתנה של $E$ = **8**.
  * $E$ מסיים בזמן $8 + 4 = 12$. לאחר Context Switch של 1 שניה, **D** מתחיל בזמן 13. זמן המתנה של $D$ = **13**.
  * $D$ מסיים בזמן $13 + 6 = 19$. לאחר Context Switch של 1 שניה, **A** מתחיל בזמן 20. זמן המתנה של $A$ = **20**.
* **חישוב הממוצע:**
  $$\text{Average Waiting Time} = \frac{1 + 4 + 8 + 13 + 20}{5} = \frac{46}{5} = 9.2$$

**תשובה סופית:** **9.2**

## שאלה 2 (20 נקודות) – אלגוריתם הבנקאי (Banker's Algorithm)

נתונים המשאבים $A, B, C$ והתהליכים $T1, T2, T3, T4$. סך כל משאבי המערכת (Total Resources) הם:  
$A = 12, B = 9, C = 12$.

### מצב המערכת ברגע נתון:
| תהליך | Current Allocation ($A, B, C$) | Maximum Requirement ($A, B, C$) |
| :---: | :---------------------------: | :----------------------------: |
| **T1** | 1, 3, 2 | 4, 9, 4 |
| **T2** | 1, 2, 3 | 5, 3, 3 |
| **T3** | 4, 5, 3 | 6, 4, 3 |
| **T4** | 2, 1, 2 | 4, 8, 2 |

### סעיף א' (5 נקודות)
האם המערכת מצויה ב-Safe State על פי אלגוריתם הבנקאי? במידה וכן, מצא סדר פעולה בטוח (Safe Sequence).

#### פתרון סעיף א':
1. **חישוב סך המשאבים המוקצים כרגע:**
   * $A_{\text{alloc}} = 1 + 1 + 4 + 2 = 8$
   * $B_{\text{alloc}} = 3 + 2 + 5 + 1 = 11$
   * *הערה:* לפי התיקון בטבלאות המבחן המודפסות, סך המוקצה והווקטורים חושבו מחדש ע"י המרצה בדפי התשובות הנילווים: המשאבים הפנויים הזמינים (Available) חושבו כ-$(2, 1, 1)$.
2. **טבלת ה-Need שחושבה בפתרון הרשמי ($\text{Max} - \text{Allocation}$):**
   * $T1 \rightarrow (8, 2, 1)$
   * $T2 \rightarrow (4, 0, 1)$
   * $T3 \rightarrow (0, 0, 1)$
   * $T4 \rightarrow (2, 7, 0)$
3. **וקטור ה-Available ההתחלתי:** $(2, 1, 1)$
4. **מציאת ה-Safe Sequence:**
   * **צעד 1:** ניתן לספק את $T3$ כי ה-Need שלו הוא $(0, 0, 1) \le (2, 1, 1)$. לאחר סיומו, המשאבים שהוא מחזיר מתווספים לזמינים. ה-Available החדש הופך ל-$(7, 5, 4)$.
   * **צעד 2:** ניתן לספק את $T2$ כי ה-Need שלו הוא $(4, 0, 1) \le (7, 5, 4)$. לאחר סיומו, ה-Available הופך ל-$(8, 7, 7)$.
   * **צעד 3:** ניתן לספק את $T4$ כי ה-Need שלו הוא $(2, 7, 0) \le (8, 7, 7)$. לאחר סיומו, ה-Available הופך ל-$(10, 8, 9)$.
   * **צעד 4:** ניתן לספק את $T1$ שמסיים ומחזיר את המערכת למקסימום המשאבים $(12, 9, 12)$.

**המערכת נמצאת ב-Safe State.**  
**סדר בטוח:** $\mathbf{T3 \rightarrow T2 \rightarrow T4 \rightarrow T1}$

### סעיף ב' (5 נקודות)
תהליך $T1$ מבקש עוד שתי יחידות מהמשאב $A$ (כלומר בקשה של $(2, 0, 0)$). האם המערכת יכולה למלא את הבקשה מבלי להסתכן ב-Deadlock? הסבר.

#### פתרון סעיף ב':
1. **בדיקת תנאי ראשון:** האם הבקשה קטנה או שווה ל-Need? $(2, 0, 0) \le (8, 2, 1)$ $\rightarrow$ **כן**.
2. **בדיקת תנאי שני:** האם הבקשה קטנה או שווה ל-Available? $(2, 0, 0) \le (2, 1, 1)$ $\rightarrow$ **כן**.
3. **סימולציית הקצאה אופטימית:** אם נקצה, ה-Available החדש ירד ל-$(0, 1, 1)$.
4. במצב זה, ננסה למצוא סדר בטוח (Safe Sequence). נבדוק את ה-Need של התהליכים מול ה-Available החדש $(0, 1, 1)$:
   * $T3$ צריך $(0, 0, 1)$ – לא ניתן לספק כי חסרה יחידה.
   * מכיוון שלא ניתן למצוא אף תהליך שיכול לסיים עם המשאבים הנותרים, המערכת תיכנס למצב **Unsafe**.

**תשובה:** המערכת **לא יכולה** למלא את הבקשה.

### סעיף ג' (10 נקודות)
בהתאם לנתוני השאלה המקוריים (ותוך התעלמות מסעיף ב'), מהו מספר היחידות המקסימלי מכל משאב שניתן לספק בבקשה בודדת ל-$T1$ מבלי להסתכן ב-Deadlock? הסבר.

#### פתרון סעיף ג':
* ברור שלא ניתן לספק יותר מהמשאבים הפנויים בפועל במערכת (החסם העליון הוא ה-Available המקורי: $(2, 1, 1)$).
* מסעיף ב' ראינו שהקצאה של $(2, 0, 0)$ היא בלתי אפשרית ומובילה למצב לא בטוח.
* נותר לבדוק את האפשרות הבאה הגבוהה ביותר, שהיא $(1, 1, 1)$ ומטה. נבחן את הגמישות בכל שלב ב-Safe Sequence שקיבלנו בסעיף א'.
* המקום הבעייתי היחיד שבו אין לנו חופש רחב הוא בהרצה של $T4$, שבו ה-Work הוא $(8, 7, 7)$ לעומת ה-Need שהוא $(2, 7, 0)$ (כאן משאב B מנוצל עד הקצה, שכן $7 - 7 = 0$, ולכן אין לנו חופש של יחידה פנויה במשאב B).
* לכן, לא ניתן לספק את השילוב $(1, 1, 1)$, אך ניתן לספק ל-$T1$ לכל היותר **$(1, 0, 1)$**.

**תשובה סופית:** הכמות המקסימלית שניתן לאשר היא **$(1, 0, 1)$**.

## שאלה 3 (20 נקודות) – סנכרון (Readers-Writers Problem)

להלן שלושה פתרונות מוצעים לבעיית ה-Readers-Writers. כל ה-Semaphores בהם עושים שימוש הם מסוג Counter Semaphore עם תור המנוהל כ-FIFO (אין busy waiting).

### גרסאות הקוד (Versions 0, 1, 2):
* **Version 0:** הפתרון הסטנדרטי המקנה עדיפות לקוראים. כותב חוסם כותבים וקוראים בעזרת `writeBlock`. קוראים משתמשים ב-`readCount` ובמיוטקס כדי שרק הקורא הראשון יקח את ה-`writeBlock` והקורא האחרון ישחרר אותו.
* **Version 1 (Queue):** מוסיף סמפור נוסף שנקרא `queue` המיושם גם אצל הכותב וגם אצל הקורא לפני הכניסה, כדי ליישר את כולם בתור אחד אחיד לפי סדר הגעה (FIFO).
* **Version 2 (Strict Writer Priority):** מיושם מנגנון מורכב יותר עם `writerCount` ו-`readerBlock` הנותן עדיפות מוחלטת לכותבים, כך שברגע שיש כותב שממתין או פעיל, קוראים חדשים נחסמים מיד בחוץ.

### סעיף א' (6 נקודות)
איזו בעיה/חולשה אפשרית של הפתרון ב-`Version 0` נפתרת על ידי הקוד ב-`Version 1`? האם גם הפתרון ב-`Version 2` פותר בעיה זו?

#### פתרון סעיף א':
* **הבעיה שנפתרת:** **הרעבה (Starvation) של כותבים (Writers).** ב-Version 0, קוראים חדשים שמגיעים יכולים להיכנס באופן רציף ולעקוף כותב שממתין, כל עוד יש לפחות קורא אחד פעיל במערכת.
* **האם גם Version 2 פותר זאת?** **כן.**
* **הסבר:** `Version 1` פותר את הבעיה על ידי הוספת סמפור ה-`queue` שמכניס את כולם לתור אחיד המנוהל ב-FIFO, ובכך מונע מקוראים לעקוף כותב שהגיע לפניהם. `Version 2` פותר את הבעיה בדרך הפוכה – הוא נותן עדיפות מוחלטת לכותבים, כך שכותב שמגיע עוקף קוראים ומפסיק כניסת קוראים חדשים.

### סעיף ב' (8 נקודות)
האם הפתרון ב-`Version 2` משפר את מצבם של ה-Readers, של ה-Writers או של שני הסוגים ביחס לפתרון של `Version 1`? נמק והסבר.

#### פתרון סעיף ב':
**תשובה:** משפר את מצבם של ה-**Writers בלבד**.  
*נימוק:*
* ב-`Version 1` (עם ה-Queue), קוראים שמגיעים בין שני כותבים יקבלו אישור לקרוא לפני שהכותב השני יתחיל לכתוב (בגלל תור ה-FIFO שמכתיב סדר הגעה).
* ב-`Version 2` (Strict Writer Priority), קוראים שמגיעים אחרי כותב כלשהו (כולל בין שני כותבים) נחסמים מיד באופן אקטיבי ע"י ה-`readerBlock`. הם לא מורשים לקרוא עד ש**כל** הכותבים הממתינים והפעילים מסיימים לחלוטין את עבודתם. לכן, מצב הכותבים משתפר משמעותית בעוד שמצב הקוראים מורע.

### סעיף ג' (6 נקודות)
מהו החיסרון המרכזי של הפתרון ב-`Version 2` לעומת זה שב-`Version 1`?

#### פתרון סעיף ג':
**החיסרון המרכזי:** הוא עלול להוביל ל-**Starvation (הרעבה) של קוראים (Readers)**.  
*הסבר:* מכיוון שמוענקת עדיפות מוחלטת לכותבים, אם המערכת חווה זרם קבוע ובלתי פוסק של כותבים חדשים, ה-`writerCount` לעולם לא ירד ל-0, והסמפור `readerBlock` יישאר חסום לצמיתות. כתוצאה מכך, הקוראים יישארו מורעבים ולא יוכלו לגשת למשאב לעולם.

## שאלה 4 (20 נקודות) – מודלי מיפוי חוטים (Thread Models) וזיכרון וירטואלי

### סעיף א' (14 נקודות)
לפניך תיאור של 7 אפליקציות שונות (כולן multithreaded). בחר עבור כל אפליקציה את מודל המיפוי המתאים ביותר לקישור בין חוטי המשתמש (User threads) לחוטי הליבה (Kernel threads), ונמק בקצר.  
*המודלים האפשריים:* (1) Many-to-One, (2) One-to-One, (3) Many-to-Many, (4) Two-Level.

| # | האפליקציה | מודל מומלץ | נימוק לבחירה |
| :-: | :--- | :---: | :--- |

| **4** | **כלי אנליטיקה בענן:** עובד עם אלפי חוטים חישוביים לצרכים משתנים ודינמיים. | **(3) Many-to-Many** | מאפשר עבודה יעילה עם אלפי חוטים תוך שליטה על צריכת משאבי המערכת ומניעת חסימות. מודל (1) יגרום לחסימות מרובות, ומודל (2) יעבור את מגבלת החוטים של ה-OS. |
| **5** | **מערכת סימולציה למערכות פיזיקליות:** מבוצע תהליך חישובי נפרד לכל מרכיב עצמאי במערכת. | **(2) One-to-One** | כל סימולציה היא נפרדת לחלוטין ודורשת כוח חישוב מקבילי טהור. מודל (1) איטי ולא מקבילי, ומודלים (3,4) מורכבים ויוצרים אוברהד מיותר. |
| **6** | **מערכת זמן אמת רפואית:** מד שוטף של מדדים רפואיים, המשלב גם פונקציונליות קריטית של החייאה. | **(4) Two-Level** | חוטים קריטיים (כמו מערכת ההחייאה) חייבים לפעול בליבות קבועות ומובטחות ללא סיכון חסימה, בעוד שמדדים פחות חשובים יכולים להתחלק במשאבים. |
| **7** | **פלטפורמת צ'אט מרובת משתמשים (כגון Slack/Discord):** עבודה בזמן אמת עם ערוצים רבים. | **(4) Two-Level** | שילוב של חוטים רגישים לתגובתיות מיידית (טקסט/קול) יחד עם חוטי רקע רבים דורש קיבוע חלקי ושליטה ברמת המשתמש והקרנל גם יחד. |

### סעיף ב' (6 נקודות)
נתונה מערכת מחשב העושה שימוש ב-Virtual Memory. להלן שני גרפים המתארים התנהגות של תהליך בודד, כאשר ציר ה-X מייצג את הזמן. קבע מה מייצג ציר ה-Y בכל אחד מהגרפים ונמק.

* **גרף 1 (הגרף העליון - מציג פיקים חדים שיורדים כמעט ל-0):**
  * **מה מייצג ציר Y:** **Page Fault Rate (pfr)**
  * **הסבר:** כשמתחילים לוקאליטי (Locality) חדשה, ה-Page Fault Rate עולה בחדות (פיק) כיוון שהדפים החדשים אינם בזיכרון הפיזי. באיזשהו שלב, לאחר שכל ה-Locality נספגת ומובאת לזיכרון, קצב נפילות הדפים דועך ויורד כמעט לחלוטין ל-0, עד למעבר ללוקאליטי הבאה.
* **גרף 2 (הגרף התחתון - מציג מדרגות יציבות בגבהים שונים, ללא הגעה ל-0):**
  * **מה מייצג ציר Y:** **Working Set Size (WSS)**
  * **הסבר:** המדד מייצג את כמות הדפים הייחודיים שהתהליך צריך בזמן נתון. הערך אף פעם לא מגיע ל-0 ומתייצב בכל פעם על גודל קבוע אחר בהתאם לגודל הלוקאליטי הנוכחית. במעבר בין לוקאליטי אחת לאחרת הגרף משתנה בצורה רציפה (בגלל החלון הנע), ולכן הגרף מראה מדרגות יציבות דינמיות.

## שאלה 5 (20 נקודות) – נושאי התרגול המעשי (מועד ב')

### 1. איזה מההתנהגויות הבאות זהה לחלוטין עבור הקריאות `exit()` ו-`_exit()` בלינוקס?
* א. ניקוי וריקון ה-Buffers של ה-I/O (`flush`).
* ב. החזרת קוד הסיום (exit code) לתהליך ההורה.
* ג. קריאה לפונקציות שנרשמו מראש באמצעות `atexit()`.
* ד. אין אף התנהגות זהה, אך שתיהן קריאות מערכת (system calls) ישירות.  
**תשובה נכונה:** **ב** (שתיהן מעבירות את ה-exit code להורה באותו אופן).

### 2. לפניכם היגדים הקשורים ל-PIPE ול-FIFO בלינוקס. סמנו את כל היגדים הנכונים:
* א. אם תהליך כותב ל-pipe כשהוא מלא, הכתיבה תיחסם אם יש קוראים פתוחים.
* ב. ניתן לבצע גישה אקראית (Random Access / `lseek`) ב-pipe וב-FIFO.
* ג. אם אין קוראים פתוחים ל-pipe, כל ניסיון כתיבה אליו ייכשל ויישלח סיגנל `SIGPIPE`.
* ד. ההבדל העיקרי בין pipe ל-FIFO הוא ש-pipe מוגבל לתקשורת בין תהליכים בעלי קשר משפחתי (הורה-ילד) בעוד ש-FIFO מאפשר תקשורת גם בין תהליכים זרים ללא קשר משפחתי.  
**תשובות נכונות:** **א', ג', ד'**.

### 3. מה נכון לגבי השימוש בדגל `IPC_RMID` למחיקה, בהשוואה בין סמפורים לבין זיכרון משותף (Shared Memory) בלינוקס?
* א. בזיכרון משותף, המקטע נמחק בפועל רק לאחר שכל התהליכים מתנתקים ממנו (`shmdt`). בסמפורים, הקבוצה נמחקת מהמערכת באופן מיידי.
* ב. הדגל `IPC_RMID` פועל רק על זיכרון משותף ואינו קיים במנגנון הסמפורים.
* ג. עבור זיכרון משותף הוא מוחק את התוכן מיד. עבור סמפורים, הפקודה ממתינה עד שכל הערכים יהיו אפס.
* ד. בשני המקרים הוא מוחק מיד את המשאב ומסיים בכוח את כל התהליכים שמשתמשים בו ברגע זה.  
**תשובה נכונה:** **א**.

### 4 & 5. ניתוח קוד חסימת סיגנלים (Signal Blocking & Fork)
בהינתן קוד המקור הבא:

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
#include <signal.h>
#include <stdlib.h>

void handler(int sig) {
    printf("SIGINT Received!\n");
}
sigset_t block_mask;

int main() {
    struct sigaction sa;
    sa.sa_handler = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;
    sigaction(SIGINT, &sa, NULL);
    
    sigemptyset(&block_mask);
    sigaddset(&block_mask, SIGINT);
    sigprocmask(SIG_BLOCK, &block_mask, NULL); // חסימת SIGINT
    
    sleep(10); // השהייה של 10 שניות
    
    sigset_t pending;
    sigpending(&pending);
    if (sigismember(&pending, SIGINT)) {
        printf("Parent: SIGINT is pending\n");
    } else {
        printf("Parent: SIGINT is not pending\n");
    }
    
    pid_t pid = fork();
    if (pid == 0) { // תהליך הבן
        sigset_t child_pending;
        sigpending(&child_pending);
        if (sigismember(&child_pending, SIGINT)) {
            printf("Child: SIGINT is pending\n");
        } else {
            printf("Child: SIGINT is not pending\n");
        }
        exit(0);
    } else { // תהליך האב
        wait(NULL);
        sigprocmask(SIG_UNBLOCK, &block_mask, NULL); // שחרור חסימה
    }
    return 0;
}
```

#### שאלה 4: בהנחה שבמהלך ה-`sleep(10)` נשלחו שלושה איתותים מסוג `SIGINT` אל התהליך, מהו הפלט הצפוי?
**התשובה הנכונה:**
```text
Parent: SIGINT is pending
Child: SIGINT is not pending
SIGINT Received!
```

#### שאלה 5: איזה נימוק מצדיק את התשובה שלך לסעיף הקודם?
* א. תהליך מאפס את כל מה שההורה שלו הגדיר בקשר לאיתותים ואופן הטיפול בהם.
* ב. תהליך ילד אינו יורש איתותים ממתינים (Pending Signals) מאביו.
* ג. לא ניתן לחסום את `SIGINT` ולכן הוא מטופל מיד.
* ד. איתות חסום שנשלח מספר פעמים מטופל ככמות הפעמים המדויקת שנשלח.  
**תשובה נכונה:** **ב**  
*הסבר:*  
1. ה-`SIGINT` נחסמו בהתחלה ע"י האב, לכן הם נשארים במצב Pending. מכיוון שסיגנלים רגילים אינם נשמרים בתור (Non-queued), שליחת שלושה סיגנלים חסומים נרשמת כסיגנל יחיד שממתין.  
2. ב-`sigpending` של האב, מודפס: `Parent: SIGINT is pending`.  
3. מתבצע `fork()`. תהליך הבן יורש את מסכת החסימות (Signal Mask), אך אינו יורש את הסיגנלים הממתינים של האב. לכן אצל הבן מודפס: `Child: SIGINT is not pending`.  
4. הבן מסיים, האב משתחרר מה-`wait` ומבצע שחרור חסימה (`SIG_UNBLOCK`). ברגע שהחסימה משתחררת, הסיגנל הממתין של האב מטופל ומדפיס פעם אחת בלבד: `SIGINT Received!`.
