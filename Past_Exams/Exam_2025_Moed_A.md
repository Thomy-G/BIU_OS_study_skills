## שאלה 1 (20 נקודות) – תזמון מעבד (FCFS)

במחשב של אלי יש חמישה תהליכים $A, B, C, D$ ו-$E$ המגיעים בסדר זה ($A$ ראשון, $B$ שני, $C$ שלישי, $D$ רביעי ו-$E$ חמישי) באותו הזמן עם בקשות CPU Bursts כמפורט בטבלה להלן.  
חשב את ה-Average Waiting Time במידה וה-Short Term Scheduler עובד לפי **First Come First Served (FCFS)**.

### נתוני ה-CPU Burst:
| תהליך | CPU Burst |
| :---: | :-------: |
| A | 7 |
| B | 2 |
| C | 3 |
| D | 6 |
| E | 4 |

### פתרון וחישוב:
* מסדרים את התהליכים לפי סדר הגעתם ($A \rightarrow B \rightarrow C \rightarrow D \rightarrow E$).
* **זמני ההמתנה (Waiting Times) עבור כל תהליך:**
  * $A$: משך המתנה = $0$
  * $B$: משך המתנה = $7$
  * $C$: משך המתנה = $7 + 2 = 9$
  * $D$: משך המתנה = $9 + 3 = 12$
  * $E$: משך המתנה = $12 + 6 = 18$
* **חישוב הממוצע:**
  $$\text{Average Waiting Time} = \frac{0 + 7 + 9 + 12 + 18}{5} = \frac{46}{5} = 9.2$$

**תשובה סופית:** **9.2**

## שאלה 2 (20 נקודות) – ניהול זיכרון ומערכות קבצים

בעת שהתכונן לבחינה במערכות הפעלה, עלה במוחו של עמרי רעיון לחיסול ה-Page Tables במנגנון ניהול הזיכרון. לשיטתו, עדיין ניתן יהיה להשתמש ב-Frames בגודל קבוע עבור הזיכרון הפיזי, אולם כחלופה ל-Page Table עצמו, מציע עמרי להשתמש באחת משיטות הקצאת הזיכרון המשמשות לקבצים.  
**הנחות יסוד:**
* זמן גישה לזיכרון (Memory Access Time) הוא $1 \text{ ns}$.
* אין להניח קיום של TLB.
* ניתן להניח שהמצביע הראשון של התהליך (בשיטות Contiguous ו-Linked – ה-Frame בו נמצא ה-Page הראשון; בשיטת Indexed – ה-Frame בו מאוכסנים האינדקסים) נמצא ברגיסטר, קרי חילוץ הערך שלו לא דורש נגיעה בזיכרון.

### סעיף א' (15 נקודות)
חשב את ה-EAT (Effective Access Time) עבור גישה למקום $d$ (offset) בדף $p$ (page) כאשר עובדים בשיטה של עמרי באמצעות שימוש בשיטות הבאות:

1. **Contiguous Allocation:** כל ה-Pages מאוכסנים בקבוצה רציפה של Frames בזיכרון.
2. **Linked Allocation:** בהנחה שאין FAT, והכתובת הלוגית לוקחת כבר בחשבון את המקום שתופס המצביע ל-Frame הבא.
3. **Indexed Allocation:** בהנחה שגודל ה-Page הוא $4\text{KB}$ ומרחב הזיכרון מיוצג באמצעות 4 Bytes.

#### פתרון סעיף א':
* **Contiguous Allocation:** **זמן גישה של 1**.  
  *נימוק:* מכיוון שההקצאה רציפה והמצביע לתחילת הבלוק נמצא ברגיסטר, אנו יודעים לחשב ישירות את הכתובת הפיזית ללא גישות ביניים לזיכרון, ולכן נדרשת רק הגישה הסופית לנתון עצמו.
* **Linked Allocation:** **זמן גישה של $p$** (או **$p+1$**).  
  *נימוק:* כדי להגיע לדף $p$, עלינו לקרוא $p-1$ מצביעים בתוך $p-1$ הפריימים הראשונים, ולאחר מכן לבצע את הגישה הסופית לנתון (סה"כ $p-1 + 1 = p$). היו סטודנטים שספרו את ה-Pages וקיבלו $p+1$, ותשובתם התקבלה באופן מלא.
* **Indexed Allocation:** 
  * חישוב גודל פריים אינדקסים: גודל דף הוא $4\text{KB} = 4096 \text{ Bytes}$. כל מצביע תופס $4 \text{ Bytes}$, לכן כל Index Frame מכיל:
    $$\frac{4096}{4} = 1024 \text{ מצביעים}$$
  * **אם $p < 1024$:** **זמן הגישה הוא 2**. (גישה אחת לבלוק האינדקסים כדי לשלוף את ה-Frame הפיזי, וגישה שנייה לנתון עצמו).
  * **אם $p \ge 1024$:** מתווספות $\lfloor p / 1024 \rfloor$ נגיעות זיכרון נוספות (עבור מעבר בין בלוקי אינדקסים משורשרים) כדי להגיע ל-Index Frame המתאים.
  *(הערה: מי שהניח 1023 מצביעים מתוך מחשבה שהמצביע לאינדקס הבא לוקח מקום בתוך הדף – קיבל ניקוד מלא).*

### סעיף ב' (5 נקודות)
האם וכיצד היה משתפר ה-EAT ב-Linked Allocation במידה והיינו משתמשים ב-FAT (File Allocation Table)? (ניתן לספק הסבר כללי ואין צורך לחשב).

#### פתרון סעיף ב':
**התשובה:** **לא היה משתנה.**  
*הסבר:* ה-FAT עצמה יושבת בזיכרון הראשי. לכן, גם אם היינו נעים על גבי טבלת ה-FAT במקום לשרשר בתוך הפריימים עצמם, עדיין היינו נדרשים לבצע את אותה כמות של נגיעות זיכרון (התקדמות על גבי הזיכרון), ולכן ה-EAT לא היה משתפר.

## שאלה 3 (20 נקודות) – מניעת חסימות (Deadlock Proofs)

נתונה מערכת עם $m$ יחידות זהות של משאב מסוים ו-$n$ תהליכים. כל תהליך יכול לבקש או לשחרר יחידה אחת בכל פעם, ויש לו צורך מקסימלי של בין $1$ ל-$m$ יחידות משאב.  
ידוע כי סכום כל הצרכים המקסימליים (ה-Max Need) של כל התהליכים ביחד קטן מ-$m+n$, כלומר:
$$\sum_{i=1}^{n} Max\_i < m + n$$

להלן ניסיונות להוכיח האם במערכת יכול להתרחש Deadlock או שבהכרח לא יכול להתרחש Deadlock. קבע עבור כל הוכחה אם היא **נכונה** או **שגויה**. במידה והיא שגויה, נמק במשפט אחד.  
*(שימו לב: נימוק שגוי להוכחה לא נכונה לא מזכה בנקודות כלל).*

* **Proof A:** נניח Deadlock קורה. אזי כל $m$ המשאבים נמצאים בשימוש. מכיוון ש-$\sum Max\_i < m+n$ הרי שבהכרח $\sum Need\_i = \sum Max\_i - m < n$. כלומר לפחות תהליך אחד חייב להחזיק יותר ממשאב אחד וברגע שאותו תהליך ישחרר את המשאבים שהוא מחזיק יוכלו תהליכים אחרים להתקדם וכו'. מכאן ש-Deadlock אינו אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** המסקנה מבוססת על הנחה לא תקפה – החזקת יותר ממשאב אחד אינה מרמזת שתהליך יכול לשחרר אותו באופן יזום או עצמאי באמצע תהליך ה-Deadlock ללא קבלת שאר המשאבים להם הוא זקוק.

* **Proof B:** נניח שמתקיים Deadlock. במצב זה כל ה-$m$ משאבים מוקצים ($\sum Allocation\_i = m$), ומכאן: $\sum Need\_i = \sum Max\_i - m < n$. ומכאן שקיים תהליך שעבורו $Need\_i = 0$, אולם מאחר ולכמה תהליכים יש Need הגדול מאפס, יכול להיות מעגל בינם לבין עצמם. מכאן שיכול להתקיים Deadlock.  
  **קביעה:** **שגויה**  
  **נימוק:** ההוכחה סותרת את עצמה; התהליך שעבורו $Need\_i = 0$ סיים את דרישותיו, ולכן הוא יחזיר למערכת את המשאבים שהוא מחזיק ונחזור למצב התחלתי עם סכום כולל קטן יותר של Need, מה שמונע חסימה מעגלית.

* **Proof C:** במידה וכל תהליך מבקש את המקסימום האפשרי שלו, הרי שסה"כ הדרישה יכולה להגיע ל-$m+n-1$, ומכיוון שסה"כ המשאבים במערכת הוא $m$, ואין מניעה ש-$m>n$, הרי שייתכן וכל אחד מהתהליכים מחזיק כמות מסוימת של משאבים ומבקש את ה-Max שלו, והכמות שנשארה Available היא 0 (או קטנה מה-Need של כל אחד מהתהליכים). ומכאן ש-Deadlock אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** אין קשר לוגי או מתמטי בין התנאי $m>n$ להמשך הטיעון במשפט.

* **Proof D:** מכיוון שכל תהליך זקוק לכל היותר ל-$m$ יחידות, וישנן $m$ יחידות בסך הכל, כל תהליך יכול בסופו של דבר לקבל את כל מה שהוא צריך. לכן, Deadlock אינו אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** העובדה שכל תהליך יכול תיאורטית לקבל את המקסימום לבדו אינה מונעת תנאי המתנה בו-זמניים של תהליכים שונים המגדירים מצב Deadlock.

* **Proof E:** נניח שמתקיים Deadlock. אזי כל המשאבים מוקצים, כלומר $\sum Allocation\_i = m$. מכאן ש-$\sum Need\_i = \sum Max\_i - m < n$, כלומר קיים לפחות תהליך אחד שעבורו $Need\_i = 0$. תהליך זה יסיים את פעולתו וישחרר את המשאבים שהוא תופס, וכך הלאה לגבי יתר התהליכים בצורה סדרתית, מה שמוביל לסתירה ולכן Deadlock לא יכול להתרחש.  
  **קביעה:** **נכונה**

* **Proof F:** נניח שמתקיים Deadlock. במצב זה כל המשאבים מוקצים. מכאן $\sum Need\_i = \sum Max\_i - m < n$, כלומר, כל תהליך צריך פחות מיחידה אחת נוספת, כלומר אין אף תהליך שממתין למשאב. הדבר מוביל לסתירה ולכן Deadlock לא יכול להתרחש.  
  **קביעה:** **שגויה**  
  **נימוק:** ההסקה שאם הממוצע קטן מ-1 אז *כל* תהליך צריך פחות מיחידה אחת איננה נכונה מתמטית ולוגית (ייתכן שחלק צריכים 0 וחלק צריכים יותר מ-1).

* **Proof G:** נגדיר את ה-Worst Case כמצב בו כל תהליך מחזיק בכמות יחידות משאב השווה ל-Maximum שלו פחות אחד ($Allocation\_i = Max\_i - 1$). מכאן:
  $$\sum Allocation\_i = \sum (Max\_i - 1) = \sum Max\_i - n$$
  מכיוון שנתון $\sum Max\_i < m+n$, מתקיים ש-$\sum Max\_i - n < m$, ולכן $\sum Allocation\_i < m$.  
  כלומר, אפילו במצב הגרוע ביותר לא כל יחידות המשאב מוקצות לתהליכים, תמיד יש לפחות משאב אחד פנוי ($Available \ge 1$), ולכן קיים תהליך שיכול לקבל עוד יחידת משאב, לסיים ולשחרר משאבים לטובת האחרים. ומכאן ש-Deadlock לא יכול לקרות.  
  **קביעה:** **נכונה**

* **Proof H:** מכיוון שכל תהליך מבקש יחידת משאב אחת בכל פעם, המערכת תמיד תוכל להתקדם. לכן Deadlock אינו אפשרי.  
  **קביעה:** **שגויה**  
  **נימוק:** הנחה זו מניחה התקדמות מבלי להוכיח שלא תיווצר המתנה מעגלית; בקשת יחידה אחת בכל פעם כשלעצמה אינה מונעת Deadlock.

* **Proof I:** במצב בו $\sum Max\_i < m+n$ יכול להתרחש Deadlock מכיוון שכל תהליך יכול לבקש יחידת משאב אחת בלבד ולהחזיק בה. במידה וכל תהליך גם יבקש יחידה נוספת, נוצרת המתנה מעגלית ואף אחד לא יכול להמשיך.  
  **קביעה:** **שגויה**  
  **נימוק:** במצב המתואר מוחזקות $n$ יחידות ומתבקשות עוד $n$ יחידות, אך מכיוון ש-$\sum Max\_i < m+n$ ובהכרח $m > n$ לפי נתוני השאלה, יישארו משאבים פנויים במערכת שניתן להקצותם, ולכן המעגל לא ייחסם.

* **Proof J:** מכיוון ש-$\sum Max\_i < m+n$, ויש לנו $m$ יחידות מהמשאב, אז בכל עת יש מספיק משאבים כדי לספק לפחות תהליך אחד. לפיכך, Deadlock לא יכול להתרחש.  
  **קביעה:** **שגויה**  
  **נימוק:** המסקנה מניחה באופן שגוי שביקוש כולל היפותטי מבטיח זמינות של משאבים פנויים בכל רגע נתון, תוך התעלמות מאופן ההקצאה הדינמית בפועל.

## שאלה 4 (20 נקודות) – זיכרון ו-TLB (גרפים וחישובים)

נתון תהליך עם 50 דפים (Pages). המערכת בה פועל התהליך **לא עושה שימוש ב-Virtual Memory**, אבל כן עושה שימוש ב-TLB. זמן גישה לזיכרון הוא $1$, וניתן להזניח את משך החיפוש ב-TLB. סך ה-RAM Frames קטן מ-128.

### סעיף א' (10 נקודות)
השלם בצורה סכמתית את הגרפים הבאים (יש לציין ערכים רלוונטיים בצירים ואת עקומת הגרף):

1. **גרף 1:** ה-EAT (ציר Y) כפונקציה של ה-TLB Hit Ratio (ציר X, מסומן ב-$\alpha$), כאשר ה-Page Table ממומש בצורה סטנדרטית (Single-level page table).
2. **גרף 2:** יחס ה-EAT בין מימוש של Two-level page table לבין מימוש של Single-level page table, כפונקציה של ה-TLB Hit Ratio ($\alpha$).

#### פתרון סעיף א':
1. **גרף 1 (EAT עבור Single-Level):**
   * באפס פגיעות ($\alpha = 0$): ניגשים ל-TLB (פספוס), ניגשים ל-Page Table (גישה 1) ואז ל-Page בזיכרון (גישה 2) $\rightarrow$ סה"כ $2$ גישות בזיכרון.
   * בפגיעה מלאה ($\alpha = 1$): מוצאים ישר ב-TLB וניגשים רק ל-Page בזיכרון $\rightarrow$ סה"כ $1$ גישה בזיכרון.
   * הגרף הוא **קו ישר לינארי יורד** מהנקודה $(0, 2.0)$ לנקודה $(1, 1.0)$.
2. **גרף 2 (יחס ה-EAT בין Two-Level ל-Single-Level):**
   * הנוסחה ל-Two-Level: $EAT_{2} = \alpha \cdot 1 + (1-\alpha) \cdot 3 = 3 - 2\alpha$ (שכן בפספוס יש 2 גישות לטבלאות וגישה אחת לנתון).
   * הנוסחה ל-Single-Level: $EAT_{1} = \alpha \cdot 1 + (1-\alpha) \cdot 2 = 2 - \alpha$.
   * היחס הוא: $\text{Ratio}(\alpha) = \frac{3 - 2\alpha}{2 - \alpha}$.
   * עבור $\alpha = 0$: היחס הוא $\frac{3}{2} = 1.5$.
   * עבור $\alpha = 1$: היחס הוא $\frac{1}{1} = 1.0$.
   * הגרף הוא **עקומה לא לינארית קמורה כלפי מטה** (יורדת מ-1.5 ל-1.0).

### סעיף ב' (6 נקודות)
צייר את גרף ה-EAT (ציר Y) כפונקציה של ה-TLB Hit Ratio (ציר X), במידה וה-Page Table ממומש כ-**Hashed Page Table**, כאשר פונקציית ה-Hash היא השארית של חלוקה ב-10 ($p \pmod{10}$). כתוב את הביטוי והסבר את מרכיביו.

#### פתרון סעיף ב':
* **ניתוח הנתונים:** לתהליך יש 50 דפים, וטבלת ה-Hash היא בגודל 10 כניסות (חלוקה ב-10).
* בהנחה שהדפים מתפלגים בצורה אחידה, אורך כל רשימה מקושרת בכל כניסה בטבלה הוא בממוצע: $\frac{50}{10} = 5$ רשומות.
* בחיפוש ברשימה מקושרת באורך 5, בממוצע נעבור על מחציתה, כלומר $3$ רשומות (הרשומה ה-1, ה-2 וה-3 שבה נעצרים).
* עבור כל רשומה שלא מצאנו בה, אנו קוראים את ה-Page מספר ואת המצביע לרשומה הבאה (2 קריאות). ברשומה שמצאנו בה, קוראים את ה-Page ואת ה-Frame (אך לא צריך את המצביע הבא).
* סך הכל גישות ממוצעות לטבלה במקרה של TLB Miss: $7$ קריאות בזיכרון, ועוד קריאה $1$ סופית של הנתון עצמו $\rightarrow$ סה"כ **8 נגיעות בזיכרון במקרה של Miss**.
* במקרה של TLB Hit ($\alpha = 1$): נדרשת **נגיעה 1 בלבד** (הגישה לנתון).
* **הגרף:** קו לינארי ישר היורד מהנקודה $(0, 8.0)$ לנקודה $(1, 1.0)$.

### סעיף ג' (4 נקודות)
סמן את הקביעות הנכונות (ניתן לסמן כמה שרוצים או אף תשובה):
* א. אלגוריתם Shortest Job First יבטיח לכל תהליך את ה-Waiting Time המינימלי במידה וזמן ה-Context Switch זניח וכל התהליכים נכנסים ל-Ready Queue באותו זמן.
* ב. אלגוריתם Shortest Job First יבטיח לכל תהליך את ה-Waiting Time המינימלי במידה ומדובר בגרסה ה-Preemptive שלו וזמן ה-Context Switch זניח.
* ג. באלגוריתם Round Robin, זמן ההמתנה של תהליך יכול להיות גדול אם ה-Time Quantum בו עושים שימוש הוא קטן מאוד.
* ד. FCFS הוא האלגוריתם היחיד שמאפשר לתהליכים לרוץ בצורה רציפה עד לסיום ה-Time Quantum שלהם.

#### פתרון סעיף ג':
* **תשובה ג' היא הנכונה היחידה.**  
  *הסבר:* א' ו-ב' אינם נכונים כיוון ש-SJF מבטיח זמן המתנה מינימלי **בממוצע** (Average Waiting Time) ולא ברמת התהליך הבודד. ד' אינו נכון כיוון שגם אלגוריתמים כמו Priority או SJF בגרסתם ה-Non-preemptive מאפשרים לתהליך לרוץ ברציפות ללא קטע קוונטום קטוע. ג' נכון כיוון שקוונטום קטן מאוד גורם לעומס עצום של Context Switches שמעלה את זמני ההמתנה.

## שאלה 5 (20 נקודות) – נושאי התרגול המעשי

### 1. מה משותף ל-Hard Links ול-Symbolic Links (Soft Links)?
* א. שניהם יכולים להצביע על קבצים במערכות קבצים (partitions) שונות.
* ב. שניהם מפסיקים לעבוד אם הקובץ המקורי נמחק.
* ג. לשניהם יש i-node משלהם השונה מהקובץ המקורי.
* ד. את שניהם אפשר ליצור דרך ה-Command Prompt/Shell בעזרת הפקודה `ln`.  
**תשובה נכונה:** **ד** (הפקודה `ln` ללא דגלים מייצרת Hard Link, ועם הדגל `-s` מייצרת Soft Link).

### 2. אילו מהאפשרויות הבאות עשויות לשפר ביצועים בקריאה וכתיבה מהדיסק בלינוקס? (סמנו את כל התשובות הנכונות)
* א. לקרוא יותר נתונים ממה שצריך בפועל ולשמור אותם בזיכרון מקומי (Read-ahead / Caching).
* ב. לבצע פתיחה וסגירה של ה-File Descriptor מחדש בכל קריאה או כתיבה כדי לוודא שאין נעילה.
* ג. לכתוב נתונים לזיכרון מקומי תחילה, ורק לאחר מכן לרשום אותם לדיסק במרוכז (Buffering / Delayed Write).
* ד. לקרוא ולכתוב כל תו בנפרד כדי להקטין עומס על הזיכרון.  
**תשובות נכונות:** **א' ו-ג'**.

### 3. מהי הסיבה העיקרית להשתמש ב-`pthread_mutex_lock` במקום ב-`pthread_mutex_trylock`?
* א. להבטיח שתור ההמתנה יהיה קצר יותר.
* ב. לאפשר ל-Thread עבודה חלופית במקום להיחסם.
* ג. אפשרות לתפוס את המשאב על אף שהוא נעול.
* ד. לחסוך הקצאת זיכרון במנגנון Kernel-space.  
**תשובה נכונה:** **ב'** (מטרת `trylock` היא לא להיחסם אלא להחזיר קוד שגיאה מיידי אם המיוטקס נעול, מה שמאפשר לבצע עבודה חלופית, ולכן הבחירה ב-`lock` הרגיל נעשית כשאין עבודה חלופית ורוצים להמתין לחסימה בצורה יעילה).

### 4 & 5. ניתוח קוד סיגנלים וצינורות (Signals & Pipes)
בהינתן קוד המקור הבא:

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
#include <signal.h>
#include <stdlib.h>

void handler_usr1(int sig) {
    printf("Handler SIGUSR1: Signal %d received in process %d\n", sig, getpid());
}
void handler_usr2(int sig) {
    printf("Handler SIGUSR2: Signal %d received in process %d\n", sig, getpid());
}
int main() {
    int pipefd[2];
    pipe(pipefd);
    signal(SIGUSR1, handler_usr1);
    
    pid_t pid1 = fork();
    if (pid1 == 0) { // בן ראשון
        close(pipefd[1]);
        char buf;
        read(pipefd[0], &buf, 1);
        close(pipefd[0]);
        kill(getpid(), SIGUSR1);
        return 0;
    }
    
    signal(SIGUSR2, handler_usr2);
    pid_t pid2 = fork();
    if (pid2 == 0) { // בן שני
        close(pipefd[0]);
        kill(getppid(), SIGUSR2);
        write(pipefd[1], "V", 1);
        close(pipefd[1]);
        return 0;
    }
    
    close(pipefd[0]);
    close(pipefd[1]);
    pause();
    waitpid(pid1, NULL, 0);
    printf("Parent process %d done!\n", getpid());
    return 0;
}
```

**נתונים והנחות:**
* הורה (Parent) PID = 20.
* בן ראשון (Child 1) PID = 21.
* בן שני (Child 2) PID = 22.
* האב מגיע לשורה של ה-`pause()` לפני שהבנים מתחילים לרוץ.
* ערך הסיגנל `SIGUSR1` הוא 10, ו-`SIGUSR2` הוא 12.

#### שאלה 4: מה תהיה השורה הראשונה שתודפס בעת הרצת התוכנית?
* א. `Handler SIGUSR1: 10 received in process 20`
* ב. `Handler SIGUSR1: 10 received in process 21`
* ג. `Handler SIGUSR2: 12 received in process 20`
* ד. `Handler SIGUSR2: 12 received in process 22`  
**תשובה נכונה:** **ג**  
*הסבר:* הבן הראשון (21) חסום בקריאת `read` מהצינור כי הצינור ריק. הבן השני (22) רץ מיד ושולח `SIGUSR2` לאבא שלו (20) באמצעות `kill(getppid(), SIGUSR2)`. האבא שהיה ב-`pause()` מתעורר ומריץ את ה-Handler שלו, שמדפיס את השורה של `SIGUSR2` עבור תהליך 20. רק לאחר מכן הבן השני כותב לצינור ומשחרר את הבן הראשון.

#### שאלה 5: נניח שמחליפים את SIGUSR1 בשורה 29 (אצל הבן הראשון) ב-SIGUSR2. מה תהיה השורה השנייה שתודפס?
* א. `Handler SIGUSR2: 12 received in process 21`
* ב. `Handler SIGUSR2: 12 received in process 20`
* ג. `Handler SIGUSR1: 10 received in process 20`
* ד. `Parent process 20 done!`  
**תשובה נכונה:** **ד** (בפועל, כפי שהוסבר, הדבר נובע מקריסת הבן הראשון בשל היעדר Handler מתאים ל-SIGUSR2, דבר המשחרר את ה-waitpid של האב).
